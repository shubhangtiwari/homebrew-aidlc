# Module Blueprints

Blueprints are living low-level design documents for modules. They capture contracts that code and
specs must respect: owned state, read-only paths, graph or workflow topology, integration
boundaries, and test gates.

Keep them current: whenever **`implementer`** changes something a blueprint documents — including
trivial and small work without a spec — update the blueprint in the **same PR** when needed. No
spec does not mean no blueprint maintenance.

`docs/blueprints/module-template.md` is generated by the `init-architecture` skill from the
chosen reference profile under `.ai/references/architectures/<id>/blueprint-template.md`. Run
init before authoring the first blueprint.

Use one root blueprint per module:

```text
docs/blueprints/<module>.md
```

Large internal sections may be hoisted into a tree:

```text
docs/blueprints/<module>/<section>.md
```

The root blueprint remains the index and owner of module-wide invariants.

## What belongs here

- Module purpose and scope.
- Layer map.
- Public and internal contracts.
- Owned state or durable data.
- Read-only paths.
- Workflow, graph, or message topology.
- Integration boundaries.
- Test gates and required fixtures.

## What does not belong here

- Product motivation. Use a PRD.
- Cross-cutting architecture decisions. Use an ADR.
- One-off implementation steps. Use a spec.
